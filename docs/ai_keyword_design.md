# AI 키워드 제안 대화형 UX — 상세 설계

> PigeonBrief의 핵심 차별화 기능. 사용자가 자연어로 관심 주제를 설명하면 AI가 검색 전문가 역할로 키워드와 RSS 소스를 제안하고, 사용자는 결정만 한다.

본 문서는 Phase 0 (사전 설계 및 검증)와 Phase 1 (구현)의 청사진이다. 새로운 세션에서도 이 문서만 보면 작업을 이어갈 수 있어야 한다.

---

## 1. 기능의 본질

사용자는 "내가 뭘 알고 싶은지는 알지만, 그걸 어떤 키워드로 검색해야 좋은 뉴스가 나올지는 모른다". AI는 이 갭을 메우는 **검색 전문가**여야 한다.

## 2. 핵심 설계 원칙

1. **대화형이되 짧게** — 이상적으로 3~5회 상호작용으로 완료
2. **AI는 제안자, 사용자는 결정자** — 모든 AI 출력은 수정 가능
3. **저장 전 미리보기 필수** — 잘못된 키워드로 며칠 기다리는 경험 차단
4. **콜드 스타트 OK** — 사용자 히스토리 없이도 첫 시도부터 유용
5. **재편집 가능** — 섹션 저장 후에도 AI 대화로 돌아올 수 있음

## 3. 사용자 여정 (8 step)

```
[Step 1] 주제 자연어 입력
  예: "생성형 AI가 법률 산업에 미치는 영향"
       ↓
[Step 2] AI 해석 + (필요 시) 1회 되묻기
  "구체적으로 어떤 관점에 관심 있으세요?
   ① 법률 AI 제품/스타트업
   ② 판례·규제 변화
   ③ 로펌 도입 사례
   ④ 모두 포함"
       ↓
[Step 3] AI가 키워드 후보를 카테고리화
  - 핵심 키워드
  - 관련 기업/인물
  - 관련 개념
  - 제외 키워드 (노이즈 방지)
       ↓
[Step 4] 사용자 수정
  - 체크박스 포함/제외
  - 직접 추가
  - 자연어 요청: "블록체인 관련도 추가해줘"
       ↓
[Step 5] 검색 쿼리 미리보기 (AND/OR/NOT 조합)
       ↓
[Step 6] AI가 적합 RSS 소스 추천
  - feedparser 검증 통과한 것만
  - 사용자가 직접 RSS URL 추가도 가능
       ↓
[Step 7] 미리보기: 최근 24시간 샘플 5건
  사용자가 "원하는 뉴스가 맞다"를 즉시 확인
       ↓
[Step 8] 섹션 저장 → DB 등록 → 다음 배치부터 수집
```

## 4. 프롬프트 설계 (Qwen 2.5 14B)

3개의 분리된 프롬프트로 구성. 모두 `response_format: json_object`로 강제.

### 프롬프트 A — 주제 해석 + 되묻기

**역할**: 뉴스 검색 전문가
**입력**: 사용자의 주제 설명 (자연어)
**출력**: JSON
```json
{
  "interpreted_intent": "사용자가 원하는 것을 한 문장으로 재정리",
  "needs_clarification": true,
  "clarification_question": "필요 시 1개만",
  "clarification_options": ["선택지1", "선택지2", "선택지3", "모두 포함"]
}
```
**규칙**:
- 주제가 이미 충분히 구체적이면 `needs_clarification=false`이고 question/options는 빈 값
- 되묻기는 최대 1회만
- `interpreted_intent`는 반드시 사용자 입력 언어로 작성

### 프롬프트 B — 키워드 + RSS 추천

**역할**: 검색 쿼리 빌더 + 뉴스 소스 전문가
**입력**: 확정된 주제 (Step 1 + Step 2 결과 통합)
**출력**: JSON
```json
{
  "core_keywords": ["핵심 3~5개"],
  "related_entities": ["기업/인물 3~5개"],
  "related_concepts": ["관련 개념 3~5개"],
  "exclude_keywords": ["노이즈 방지 키워드 0~5개"],
  "recommended_query": "(\"a\" OR \"b\") AND (\"c\" OR \"d\") NOT (\"e\")",
  "recommended_rss": [
    {"name": "소스명", "url": "RSS URL", "reason": "한 줄 추천 이유"}
  ]
}
```
**규칙**:
- RSS URL은 실제로 존재하는 것만. 확실하지 않으면 빈 배열 반환 (환각 절대 금지)
- `recommended_query`는 Naver/Google News 양쪽에서 작동하는 형태
- 영문 주제는 영어 키워드 위주 + 한국어 해당 키워드도 1~2개 병기
- 한국어 주제는 한국어 키워드 위주 + 영어 키워드 1~2개 병기
- `exclude_keywords`는 너무 많이 만들지 말 것 (과잉 차단 방지)

### 프롬프트 C — 자연어 수정 요청 처리

**역할**: 키워드 편집 보조
**입력**: 현재 키워드 세트 JSON + 사용자 수정 요청 (자연어)
**출력**: 프롬프트 B와 동일한 JSON 스키마 (수정된 버전)
**규칙**:
- 사용자가 명시적으로 삭제하라고 한 항목 외에는 기존 항목 유지
- 추가 요청은 적절한 카테고리에 배치

## 5. RSS 환각 방지

Qwen이 존재하지 않는 RSS URL을 만들어낼 위험이 있다. 대응:

**채택안: 실시간 검증**
- AI가 반환한 모든 RSS URL을 `feedparser.parse()`로 즉시 검증
- 파싱 성공 + `entries` 1개 이상 → 통과
- 실패 → 사용자에게 노출하지 않음
- 검증 후 통과한 RSS가 0개여도 OK (사용자가 직접 추가 가능)

구현: `backend/ai/rss_validator.py`

(참고: 화이트리스트 방식은 초기 비용이 크므로 Phase 6에서 검토)

## 6. API 스펙 (Phase 1에서 구현)

| 메서드 | 경로 | 입력 | 출력 |
|--------|------|------|------|
| POST | `/api/ai/interpret` | `{"topic": "..."}` | 프롬프트 A 출력 |
| POST | `/api/ai/suggest-keywords` | `{"topic": "...", "clarification_answer": "..."}` | 프롬프트 B 출력 (RSS 검증 후) |
| POST | `/api/ai/edit-keywords` | `{"current": {...}, "instruction": "..."}` | 프롬프트 C 출력 (RSS 검증 후) |
| POST | `/api/ai/preview` | `{"keywords": {...}, "rss_urls": [...]}` | 카드뉴스 5건 (실제 수집·요약) |

**공통 동작**:
- LLM 응답 24시간 캐싱 (입력 해시 기준)
- JSON 파싱 실패 시 1회 재시도, 그래도 실패면 사용자에게 "다시 시도" 버튼
- 모든 엔드포인트는 Clerk 인증 필요

## 9. Phase 0 — 검증 계획

**목표**: 프롬프트 단위로 품질을 검증해, 본격 구현 전 리스크를 제거한다.

### 9.1 테스트 주제 10개

대표가 제공한 실제 관심 주제 2개 + 다양성 확보용 8개.

### 9.2 검증 절차

1. `scripts/test_ai_keyword.py` 실행
2. 각 주제에 대해 프롬프트 A → B 순차 실행
3. 결과를 `data/test_results/<날짜>/` 디렉터리에 JSON으로 저장
4. 수동 평가표 작성 (아래 9.3)
5. 평가 결과 기반으로 프롬프트 수정 → 재실행 → 비교
6. 목표 도달 시 Phase 0 종료

### 9.3 평가 기준 (각 주제별 5점 척도)

| 항목 | 5점 | 1점 |
|------|-----|-----|
| JSON 파싱 성공 | 한 번에 성공 | 실패 |
| `interpreted_intent` 정확성 | 사용자 의도 완벽 재현 | 핵심 빗나감 |
| `core_keywords` 적절성 | 즉시 사용 가능 | 일반적/노이즈 많음 |
| `related_entities` 적절성 | 핵심 기업/인물 정확히 지목 | 무관하거나 환각 |
| `recommended_query` 실용성 | 그대로 검색 가능 | 구문 오류 또는 과잉 |
| RSS 환각 여부 | 0개 환각 | 1개 이상 환각 |

**합격 기준**: 10개 주제 중 8개 이상에서 평균 4점 이상.

### 9.4 Phase 0 종료 → Phase 1 착수 조건

- 위 합격 기준 도달
- 프롬프트 최종 버전이 본 문서에 반영됨
- 대표 검토 및 승인

## 10. 프롬프트 버전 이력

### v1 → v2 변경 이유 (2026-04-09 1차 검증 결과 기반)

v1 검증 결과 (11개 주제):
- JSON 파싱 100%, 평균 응답 시간 정상
- **RSS 환각률 62%** (16/26 실패) — 가장 심각
- **표면 단어 반복**: 사용자 주제의 단어를 그대로 키워드로 베끼는 패턴 (#2, #8, #11)
- **`related_entities`에 일반명사 혼입**: "한국은행", "ETF 투자", "한국 증권시장" 등
- **`exclude_keywords` 누락**: 11개 중 4개가 빈 배열
- **되묻기 0회**: 모호한 주제(#2, #8)에서도 되묻지 않음

### v2 핵심 변경

**프롬프트 A**
- 되묻기 판단 기준을 명문화 (관점 2개 이상 가능 / 범위 모호 / 시간 관점 모호)
- few-shot 예시 2개 추가 (모호 주제 → 되묻기 / 구체 주제 → 통과)

**프롬프트 B**
- "주제 단어 그대로 베끼기 금지" 원칙 명시
- "검색에 통하는 표현" 강조 (헤드라인에 등장할 만한 구체적 용어)
- `related_entities` 정의 엄격화 — "고유명사만, 모르면 빈 배열" 원칙
- `exclude_keywords` 최소 2개 강제
- RSS 환각 경고 강화 — "한국 매체 RSS는 특히 위험, 빈 배열이 가짜 URL보다 100배 낫다"
- 좋은 출력 few-shot 예시 1개 추가 (Legal Tech)

### 자동 평가 도입 (v2부터)

`scripts/test_ai_keyword.py`에 객관적 자동 평가 추가:
- core_keywords 개수 (3~5)
- related_entities 개수 + 고유명사 휴리스틱 검사
- exclude_keywords ≥ 2
- recommended_query 존재
- RSS 환각 없음 (feedparser 검증 → 모두 통과 또는 빈 배열)
- 5점 만점 자동 점수

목표: 11개 평균 자동 점수 4.0/5.0 이상.
